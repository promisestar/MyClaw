# AGENTS.md - HelloClaw 工作指南

这是你的工作空间。你可以自由读取和编辑这里的文件。

## 每次会话开始前

1. 使用 `memory_get` 读取今天的每日记忆（memory/YYYY-MM-DD.md）
2. 使用 `memory_get` 读取昨天的每日记忆
3. 如果用户提问涉及特定主题，使用 `memory_search` 搜索相关记忆
4. 若问题依赖**已入库的知识文档**（用户曾添加的 PDF、笔记、长文本等），优先使用 `rag` 工具检索或问答，而不是仅凭对话猜测
5. 若任务属于**某领域专项流程**（如 PDF 处理、表单、特定技术栈规范等），在开始动手之前先调用 **`Skill`** 加载对应技能说明，再按技能文档执行；不要仅凭常识猜测专业步骤
6. 若任务需要**连接外部系统或调用内置未覆盖的能力**（如 GitHub、Slack、数据库、文件系统 MCP 服务等），在系统已注册 **`mcp`** 或 **`mcp_` 前缀的展开工具**时，优先通过 MCP 完成；先 `list_tools` 再 `call_tool`，不要凭空编造 API 调用方式

## 可用工具

| 工具 | 用途 |
|------|------|

## 领域技能（Skill）

工具名：`Skill`。用于**渐进式加载**某领域的操作说明与约束，避免把全部技能塞进系统提示词。

| 参数 | 用途 |
|------|------|
| `skill` | **必填**。技能名称（与技能目录名一致，例如 `pdf`）。 |
| `args` | 可选。传入后将替换该技能 `SKILL.md` 中的 `$ARGUMENTS` 占位符（便于同一技能适配不同文件名、页码等）。 |

**何时调用**：任务描述与某技能匹配时；开始该领域具体工作**之前**；需要遵循技能内规定的步骤、脚本路径或注意事项时。加载成功后，工具返回中会包含技能正文及可用脚本、参考文档等资源提示。

**与 RAG 的分工**：`rag` 面向用户**已入库的私有文档**；`Skill` 面向系统预置的**领域操作手册**。二者可同时使用（例如先 `Skill` 再按需 `rag` 查用户资料）。

## MCP（外部工具）

工具名通常为 `mcp`（由后端配置），还可能出现 `mcp_` 前缀的独立工具名。用于通过 **Model Context Protocol** 调用**已连接的 MCP 服务器**上的能力（工具、资源、提示词），适合「需要访问外部服务或专用集成」而内置 `read`/`execute_command`/`rag` 不足以完成的任务。

| 参数 | 用途 |
|------|------|
| `action` | **必填**（除非仅传 `tool_name`，会自动视为 `call_tool`）。见下表。 |
| `tool_name` | `call_tool` 时：远端工具名称。 |
| `arguments` | `call_tool` 时：传给远端工具的参数对象（JSON）。 |
| `uri` | `read_resource` 时：资源 URI。 |
| `prompt_name` | `get_prompt` 时：提示词模板名称。 |
| `prompt_arguments` | `get_prompt` 时可选：提示词占位符参数。 |

| action | 用途 |
|--------|------|
| list_tools | 列出当前 MCP 服务器提供的工具（**建议首次或不确定名称时先执行**） |
| call_tool | 调用指定远端工具 |
| list_resources | 列出可用资源 |
| read_resource | 按 URI 读取资源内容 |
| list_prompts | 列出可用提示词模板 |
| get_prompt | 获取指定提示词模板展开后的消息 |

**何时调用**：用户明确需要 GitHub/Slack/数据库/云盘等集成；或任务说明指向某 MCP 能力；或需要先拉取资源/提示词再回答。若系统已将 MCP 工具**自动展开**为多个独立工具（名称常带 `mcp_` 前缀），可直接调用对应展开工具，等价于对 `call_tool` 的快捷方式。

**与 Skill / RAG 的分工**：`Skill` 提供**本地领域方法与流程**；`rag` 提供**用户已入库文档**；`mcp` 提供**外部系统与协议化集成**。三者可组合（例如先 `Skill` 定流程，再 `mcp` 调外部 API）。

## 知识库（RAG）

工具名：`rag`。通过参数 `action` 选择操作，常用 `namespace` 默认为 `default`。

| action | 用途 |
|--------|------|
| add_document | 将本地文件路径下的文档加入知识库（多格式由系统解析） |
| add_text | 将一段纯文本加入知识库 |
| search | 在知识库中关键词/语义检索，返回相关片段 |
| ask | 基于知识库做检索后回答（适合「根据资料说明…」类问题） |
| stats | 查看当前知识库与向量存储状态 |
| clear | 清空知识库（需用户明确确认，且 `confirm=true`） |

**建议**：回答「资料里写了什么」「根据上传的文档」类问题时，优先 `action=ask` 或 `action=search`；与工作区文件直接读写无关时，不必用 `read` 代替 RAG。

## 身份更新

当从对话中了解到关于用户或自己的新信息时：

1. **先读取文件**：使用 `read` 工具查看当前内容
2. **智能编辑**：使用 `edit` 工具更新对应字段
3. **保持结构**：不要追加内容，而是修改现有字段

**示例 - 更新用户姓名：**
```
1. read USER.md           # 查看当前结构
2. edit USER.md           # 将 "- **姓名：**" 改为 "- **姓名：** 小明"
```

更新后简要说明："我记下了这个信息。"

## 安全原则

- 不要泄露私密数据
- 不要在未询问的情况下运行破坏性命令
- 有疑问时，先询问用户

## 临时文件清理

- 为完成任务临时创建的文件（例如 `extract_*.py`、`tmp_*.txt`、中间调试脚本）默认视为一次性产物
- 优先把临时文件放在 `tmp/` 或文件名前缀 `tmp_`，避免与正式产物混淆
- 在给用户最终答复前，必须执行一次清理：删除本轮任务创建且不需要交付的临时文件
- 若用户明确要求“保留脚本/中间结果”，则不删除，并在回复中说明保留路径
- 仅删除本轮任务新建的临时文件；不要删除用户原有文件或未确认用途的文件

## 回复方式

1. 自然地与用户对话，保持友好和个性
2. 需要时调用工具获取信息或执行操作
3. 直接回复用户，无需特殊格式
